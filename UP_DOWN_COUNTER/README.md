# MOD-12 Up/Down Counter Verification (SystemVerilog)

This project implements a SystemVerilog verification environment for a MOD-12 up/down counter.
The design is verified using a UVM-based testbench, reference model, scoreboard, and
SystemVerilog Assertions (SVA).

The goal of the project is to verify that the DUT behaves correctly for reset,
load, increment, and decrement operations.

## Verification Architecture

The DUT is verified using the following components:

- Sequence / Sequencer : Generates transactions for the counter operations.

- Driver : Drives stimulus to the DUT through the interface.

- Monitor : Observes DUT inputs and outputs and forwards them to the scoreboard.

- Reference Model : Generates the expected counter output based on the inputs.

- Scoreboard : Compares the DUT output with the expected output generated by the reference model.

---

## Assertion Checks

SystemVerilog Assertions are used to verify key counter properties:

- Correct reset behavior
- Correct load operation
- Increment wrap condition (11 -> 0)
- Decrement wrap condition (0 -> 11)
- Proper up/down counting behavior

Assertions help detect incorrect DUT behavior immediately during simulation.

## Synchronization Logic (Scoreboard Alignment)

A critical part of the verification environment is maintaining proper
synchronization between the DUT output and the expected output
generated by the reference model.

At the start of simulation:

- The scoreboard expected output (exp_out) is initialized to 0
- The DUT is reset, which also sets the counter output to 0

Because both values start at 0, the first comparison naturally matches.

After this initial alignment, the scoreboard follows this order:

1. Compare DUT output with the current expected output
2. Update the expected output using the new inputs
3. Use the updated expected value for the next comparison

Conceptually:

compare current DUT output with current expected output
update expected output using current inputs

This order ensures the reference model stays synchronized with the
DUT timing.

If the expected value were updated before comparison, the reference
model would move one cycle ahead and produce false mismatches.

## How to Read the Simulation Output

When reading the simulation logs, the order of messages may look confusing.
The INPUT monitor and OUTPUT monitor do not always appear in strict order.
This happens because monitors are triggered on the same clock event and the
simulator may schedule them in any order.

### Important rule:

- DUT output always corresponds to the previous cycle input.

### Cycle relationship:
```
Cycle N     -> input applied
Cycle N+1   -> output produced
```
So when reading logs:

OUTPUT(t) must be compared with INPUT(t-1)

---

### Example :
```bash
Time = 115

OUTPUT MONITOR
data_out = 8

INPUT MONITOR
data_in = 2
load = 1
mode = 1
```
### Interpretation:

Output at time 115 was generated by the input applied at time 105.
The input at time 115 will affect the output at time 125.

---

### Next Cycle :
``` bash
Previous input (115):
data_in = 2
load = 1

RTL behavior:

if(load)
    data_out <= data_in;

Time = 125
data_out = 2

Expected = 2
Observed = 2
```

## Virtual Sequencer Based Sequence Coordination

To coordinate multiple stimulus sequences, a virtual sequencer and
virtual sequence were introduced.

The virtual sequence starts multiple sequences that exercise different
counter behaviors:

- Reset sequence
- Load sequence
- Up-count sequence
- Down-count sequence

Using a virtual sequencer allows centralized control of stimulus
generation and makes the verification environment scalable.

This structure is commonly used in UVM environments when multiple
sequences need to be coordinated through a single sequencer.

---

## Sequence Arbitration

Different UVM arbitration mechanisms are used to control the execution
order of sequences.

Reset sequence uses grab() to ensure reset operations execute first.

Load sequence uses lock() to obtain exclusive control of the sequencer
during load transactions.

Up-count and down-count sequences execute afterward and are scheduled
by the sequencer arbitration.

This guarantees the following logical stimulus order:

Reset -> Load -> Counting Operations

## Interface Based DUT Communication

An interface is used to connect the testbench and the DUT.

The interface contains:

- DUT input signals
- DUT output signals
- Driver clocking block
- Monitor clocking block

Clocking blocks ensure that signal driving and sampling occur at the
correct clock edge, avoiding race conditions between the DUT and the
verification environment.

---

## Driver and Monitor Architecture

The verification environment uses separate monitors for inputs and
outputs.

INPUT MONITOR

Observes all signals driven to the DUT and sends the captured
transactions to the scoreboard.

OUTPUT MONITOR

Observes the DUT output signals and forwards the sampled values to the
scoreboard.

Separating input and output monitoring simplifies debugging and allows
clear tracking of stimulus and DUT responses.

---

## Reference Model

The scoreboard includes a reference model that generates the expected
counter value based on the observed input transactions.

Reference model behavior:
```
Reset
counter = 0

Load
counter = data_in

Mode = 1
counter increments
11 -> 0 wrap condition

Mode = 0
counter decrements
0 -> 11 wrap condition
```
The expected value generated by the reference model is compared
against the DUT output to verify correctness.
